# Complete Upload 면접 Q&A (정합성/트랜잭션/배포/트래픽)

## 문서 목적

본 문서는 리팩토링 결과를 바탕으로 면접에서 자주 나오는 질문과 꼬리질문을 정리한다.

- 범위: 정합성, 트랜잭션, 배포, 트래픽/성능
- 구성: 기본 질문 -> 꼬리질문(딥 다이브) -> 모범 답변
- 원칙: 이론 설명보다 운영/장애 대응 관점까지 포함

---

## 1) 정합성 / 트랜잭션 Q&A

### Q1. 동일 `videoId`에 대해 `complete-upload`가 동시에 100번 들어오면 어떻게 1회 처리만 보장하나요?

#### 꼬리질문
- Redis 락이 만료되면 중복 처리되는 것 아닌가요?
- 왜 Redis만으로 안 하고 DB 락까지 쓰나요?
- 락이 걸린 상태에서 timeout이 나면 롤백/재시도는 어떻게 보장하나요?

#### 답변
- 1차로 Redis 락(`SET NX EX`)이 폭주를 줄이고, 최종 보장은 DB `PESSIMISTIC_WRITE` + 상태전이 체크로 한다.
- Redis 락 TTL 만료는 성능 문제이지 정합성 문제로 이어지지 않도록 DB 락/unique 제약을 최종 보루로 둔다.
- 트랜잭션 타임아웃/커넥션 타임아웃을 짧게 두고 실패는 빠르게 반환한다. 재시도는 idempotency 키로 안전하게 처리한다.

---

### Q2. 크레딧 차감 정합성은 어떻게 보장했나요?

#### 꼬리질문
- 왜 optimistic lock이 아니라 pessimistic lock을 택했나요?
- 차감 히스토리만 저장되고 실제 멤버 크레딧은 반영 안 되는 문제는 어떻게 막나요?
- 데드락 가능성은 없나요?

#### 답변
- 금전성 도메인은 충돌 시 재시도 비용이 커서 `Member` 행을 비관락으로 가져와 직렬화하는 전략을 택했다.
- 차감은 트랜잭션 안에서 락 조회한 `Member`에 반영하고, `credit_history`에 unique `deduction_key`를 둬 중복 차감을 방지한다.
- 데드락은 잠금 순서를 고정(영상 -> 멤버 -> 히스토리)하고 트랜잭션 구간을 짧게 유지해 완화한다.

---

### Q3. 실패 웹훅이 중복 호출되면 환불이 여러 번 되지 않나요?

#### 꼬리질문
- AI 서버가 at-least-once로 보낸다면?
- 네트워크 타임아웃으로 같은 webhook이 재전송되면?
- 환불 키 충돌 시 응답은 어떻게 하나요?

#### 답변
- 환불은 `VIDEO_PROCESSING_REFUND:{videoId}` 키를 사용해 멱등 처리한다.
- 중복 호출 시 기존 환불 레코드를 반환하고 추가 크레딧 반영은 일어나지 않는다.
- 이것은 분산 환경에서 exactly-once 대신 idempotent side-effect를 보장하는 표준 패턴이다.

---

### Q4. Outbox를 왜 도입했나요?

#### 꼬리질문
- 기존 `AFTER_COMMIT` 이벤트 리스너로 충분하지 않나요?
- Outbox도 결국 실패할 수 있는데 무슨 차이죠?
- 순서 보장은 어떻게 하나요?

#### 답변
- 커밋 직후 프로세스 다운 시 리스너 기반은 메시지 유실 창이 생긴다. Outbox는 발행 의도를 DB 트랜잭션에 같이 저장한다.
- Outbox 퍼블리셔가 실패해도 row가 남아 재시도 가능하고, 실패 누적 시 `FAILED`로 관측/복구 가능하다.
- 순서는 엄밀 보장이 필요하면 aggregate 단위 ordering key를 둬야 하고, 현재는 video/job 단위 멱등성을 우선한다.

---

### Q5. Redis가 죽으면 정합성은 정말 안전한가요?

#### 꼬리질문
- nonce replay 방지도 Redis 의존 아닌가요?
- Redis timeout으로 요청이 느려지지 않나요?
- fallback이 DB로 몰리면 DB가 터지지 않나요?

#### 답변
- Redis는 성능 계층이고 정합성 원천은 DB다. complete-upload는 DB 락/unique로 최종 보장한다.
- webhook nonce는 Redis 실패 시 DB nonce 저장소로 폴백한다.
- Redis timeout을 짧게 두고(예: 100ms), admission control로 DB 급증을 제어한다.

## 2) 배포 전략 Q&A

### Q6. 왜 바로 100% 배포하지 않고 canary를 하시나요?

#### 꼬리질문
- 기준치 없이 canary는 의미 없지 않나요?
- canary에서 어떤 메트릭을 반드시 보나요?
- 언제 즉시 롤백하나요?

#### 답변
- 본 변경은 보안/트랜잭션/메시징 핵심 경로를 건드려 blast radius가 크다. 단계 배포가 필수다.
- 핵심 메트릭은 `5xx`, `p99`, DB pool wait, lock wait, outbox pending age, webhook 401 비율이다.
- 정합성 KPI(중복 과금/중복 환불) 이상 징후가 있으면 즉시 롤백하고 원인 분석 후 재배포한다.

---

### Q7. 웹훅 서명 검증 도입 시 파트너 연동 이슈는 어떻게 줄이나요?

#### 꼬리질문
- 파트너 시간이 10분 어긋나면?
- 키 롤링 중 구버전/신버전 혼재는?
- 서명 실패가 급증하면 공격인지 버그인지 어떻게 구분하나요?

#### 답변
- 허용 오차를 운영 가능한 범위로 두고(`allowed-skew-seconds`), 파트너와 NTP 동기화를 강제한다.
- 키 롤링은 active/previous 키 동시 검증 기간을 두고 단계 전환한다.
- 실패 로그를 원인별로 분류(헤더 누락/서명 불일치/nonce 중복)해 공격 패턴과 연동 이슈를 분리한다.

---

### Q8. 배포 중 DB 스키마 변경이 필요한데 다운타임 없이 가능합니까?

#### 꼬리질문
- outbox/nonce 테이블 추가 시 락 영향은?
- 롤백 시 새 테이블은 어떻게 다루나요?
- 코드와 스키마 버전 불일치가 나면?

#### 답변
- additive migration(테이블/인덱스 추가)으로 가고 기존 경로와 호환되게 배포한다.
- 롤백은 애플리케이션만 되돌리고 테이블은 남겨도 무방하게 backward-compatible하게 설계한다.
- 스키마는 선배포, 애플리케이션은 후배포로 순서를 고정한다.

---

### Q9. 배포 후 장애가 났을 때 운영 커뮤니케이션은 어떻게 하나요?

#### 꼬리질문
- 사용자에게 뭐라고 공지하나요?
- 비즈니스/CS팀에는 어떤 수치를 전달하나요?
- 원인 불명 상태에서 재오픈 기준은?

#### 답변
- 사용자 공지는 영향 범위/임시 조치/복구 ETA를 명확히 전달한다.
- 내부에는 오류율, 처리 지연, 누적 outbox, 정합성 영향(과금 이상 여부) 같은 핵심 지표를 공유한다.
- 원인 불명이라도 재오픈은 `정합성 무결성 + 에러율 안정화 + 재발 방지 가드`가 충족될 때만 한다.

## 3) 트래픽 / 성능 / CS(시스템) Q&A

### Q10. MAU가 크고 인프라가 작을 때 왜 풀/스레드를 작게 잡나요?

#### 꼬리질문
- 크게 잡으면 더 많이 처리되는 것 아닌가요?
- CPU가 낮은데 latency가 높은 이유는?
- maxThreads를 늘리면 어떤 OS 비용이 생기나요?

#### 답변
- 작은 머신에서 큰 풀은 처리량 증가보다 대기열 증가를 먼저 만든다.
- DB/락 대기가 길면 CPU는 낮아도 tail latency가 급증한다.
- 스레드 과증가는 context switch/스택 메모리/캐시 미스 증가로 p99를 악화시킨다.

---

### Q11. 커넥션풀을 8로 줄인 이유는 무엇인가요?

#### 꼬리질문
- 30이면 더 여유로운 거 아닌가요?
- 풀 고갈 시 어떤 에러가 먼저 보이나요?
- connection-timeout을 짧게 두는 이유는?

#### 답변
- 프리티어에서 동시 SQL 30개는 DB가 소화 못해 lock wait와 timeout 폭탄으로 이어진다.
- 풀 고갈 시 acquire timeout이 먼저 보이고, 이후 요청 큐 적체로 전체 응답성이 무너진다.
- timeout을 짧게 두면 장애 전파를 줄이고, 서비스가 늦게 죽는 대신 빨리 실패해 복구 가능성이 높아진다.

---

### Q12. GC 이슈는 어떻게 보나요?

#### 꼬리질문
- Full GC가 갑자기 늘면 무엇부터 확인하나요?
- 메모리 여유가 있는데 pause가 큰 이유는?
- 로그가 GC에 미치는 영향은?

#### 답변
- 먼저 allocation rate 급증 구간, 대용량 객체 생성, 로그 폭주를 본다.
- pause는 힙 점유율뿐 아니라 객체 생존 패턴, 레퍼런스 그래프 복잡도에도 영향 받는다.
- 과도한 구조화 로그/문자열 생성은 young GC 압력을 높이므로 샘플링/레벨 조정이 필요하다.

---

### Q13. 1초 1000요청이 들어오면 어떤 순서로 방어하나요?

#### 꼬리질문
- DB 먼저 보호할 건가요, 앱 먼저 보호할 건가요?
- 429 비율이 높아지면 비즈니스 반발은?
- 어떤 요청부터 살리고 어떤 요청부터 버리나요?

#### 답변
- 입구에서 rate limit과 큐 제한으로 먼저 앱을 보호하고, 동시에 DB 커넥션/락 대기를 모니터링한다.
- 429는 실패가 아니라 보호 전략이다. 느린 전체 장애보다 빠른 부분 거절이 서비스 생존에 유리하다.
- 금전/정합성 경로를 우선하고, 비핵심/재시도 가능한 요청을 먼저 축소한다.

---

### Q14. 네트워크 파티션/일시 장애 상황에서 재시도 정책은 어떻게 잡나요?

#### 꼬리질문
- 무한 재시도하면 안 되는 이유는?
- 백오프는 어떤 근거로?
- retry storm은 어떻게 막나요?

#### 답변
- 무한 재시도는 장애를 증폭시켜 자기 자신을 DDoS한다.
- 지수 백오프와 최대 시도 횟수로 시스템 회복 시간을 준다.
- circuit breaker와 bulkhead(풀 분리)로 장애 도메인을 격리한다.

---

### Q15. DB deadlock이 발생하면 이 설계는 어떻게 대응하나요?

#### 꼬리질문
- deadlock을 완전히 없앨 수 있나요?
- 애플리케이션 재시도는 어디서?
- 운영에서 어떤 지표를 봐야 하나요?

#### 답변
- 완전 제거는 어렵고, 잠금 순서 고정/짧은 트랜잭션/작은 배치로 발생 확률을 낮춘다.
- deadlock 예외는 멱등 키 기반으로 안전한 재시도 정책을 둔다.
- DB deadlock count, lock wait timeout, 트랜잭션 시간 분포를 상시 관측한다.

---

### Q16. Redis 장애 + DB 지연이 동시에 오면 어떻게 하실 건가요?

#### 꼬리질문
- 무엇을 먼저 끄나요?
- fail-open vs fail-close 기준은?
- 정합성/가용성 충돌 시 우선순위는?

#### 답변
- 먼저 비핵심 비동기 작업을 줄이고, rate limit을 강화해 DB를 보호한다.
- 보안/금전 경로는 fail-close, 비핵심 경로는 fail-open 성격으로 분리한다.
- 이 경로에서는 정합성 우선이며, 처리량은 감소를 받아들인다.

## 4) 기본기 확인용 짧은 질문 세트

### Q17. 비관락과 낙관락 차이를 실제 서비스 맥락으로 설명해보세요.
**답변 요약**
- 낙관락: 충돌이 드문 경우 유리, 충돌 시 재시도 필요
- 비관락: 충돌이 잦거나 금전 도메인에서 유리, 대기 비용 발생
- 본 케이스는 과금 정합성이 우선이므로 비관락을 선택했다.

### Q18. 왜 unique key가 멱등성의 마지막 보루인가요?
**답변 요약**
- 애플리케이션/캐시/네트워크는 모두 실패 가능
- DB unique는 최종 커밋 시점에서 중복 side-effect를 물리적으로 차단한다.

### Q19. exactly-once가 어려운 이유는?
**답변 요약**
- 분산 시스템에서 ack 손실/중복 전송/부분 실패가 본질적으로 존재
- 그래서 현실적 해법은 at-least-once + idempotency다.

### Q20. 왜 타임아웃을 짧게 두나요?
**답변 요약**
- 긴 타임아웃은 장애 전파를 키우고 큐를 쌓아 전체 장애를 만든다.
- 짧은 타임아웃은 일부 실패를 빨리 드러내고 시스템 생존성을 높인다.

## 구현한 것
1. 멱등키(Idempotency-Key + request hash) → Redis에만 저장
2. videoId 기준 DB 비관락 → Video PENDING 상태 검증 후 Job INSERT
3. job.video_id, credit_history.deduction_key 유니크 제약
4. Outbox 패턴으로 SQS 메시지 발행 (5초 재시도)
5. 환불 웹훅 멱등 처리 (VIDEO_PROCESSING_REFUND:{videoId} 키)

# 프로젝트 2. AI 기반 영상 화질 업스케일링 서버

**한줄 요약**: 촬영한 영상을 업스케일링 서버로 요청하는 파이프라인 관리 서버

**사용 기술**: Spring Boot, JPA, Redis, MySQL, SQS

## 1. complete-upload 동시 요청으로 인한 중복 Job/중복 차감을 다층 방어(락+불변식+멱등)로 해결

### 아키텍처 문제
- 사용자가 업로드 완료 버튼을 연속 클릭하거나 네트워크 타임아웃 후 재시도하면 complete-upload가 동시에 여러 번 호출됨
- 이 API는 상태 변경(Video) + 크레딧 차감 + Job 생성 + 큐 전송을 한 번에 처리하므로, 중복 실행 시 비용/정합성 문제가 바로 발생함
- 실제 동시 요청 테스트에서 동일 videoId 기준 Job 중복 생성(평균 4.3건)과 중복 차감 위험을 확인함

### 해결 과정

#### 설계 선택 근거
- **DB 비관적 락만 사용**
    - 동시 실행은 줄일 수 있지만, 네트워크 재시도 요청을 같은 응답으로 돌려주는 API 멱등 처리 불가
- **Redis 락만 사용**
    - Redis 장애 시 최종 데이터 무결성 단독 보장 불가
- **DB 유니크 제약만 사용**
    - 최종 중복은 막지만 예외 빈발로 트랜잭션 운영 복잡도 증가

#### 최종 선택
- **DB 락 + DB 유니크 + 멱등키 + Outbox**
- 역할을 나눠서 같이 적용해, 동시 실행/재시도/부분 실패를 각각 따로 막는 방식으로 설계

### 해결 방안
- videoId 기준 DB 비관락으로 업로드 완료 로직을 직렬 처리, PENDING_UPLOAD 상태를 다시 검증해 중복 실행 차단
- job.video_id, credit_history.deduction_key 유니크 제약으로 DB 레벨에서 중복 Job 생성/중복 크레딧 차감을 최종 차단
- Idempotency-Key + videoId + request payload의 SHA-256 hash를 `video_upload_idempotency_record` 테이블에 저장해 DB 레벨에서 멱등성을 보장한다. `(member_id, video_id, idempotency_key)` 유니크 제약으로 동시 저장 경합도 차단하며, Redis 장애와 무관하게 멱등성이 유지된다. 같은 키에 다른 payload가 오면 400 충돌 오류로 처리한다.
- Redis 락 획득 시 연결 장애가 발생하면 예외를 catch해 락 미획득으로 처리하고, 이후 로직은 DB 비관락만으로 직렬화한다. Redis는 선택적 성능 레이어이고, DB 비관락 + 유니크 제약이 정합성의 최종 보장자다.
- 메시지 발행을 Outbox 패턴으로 분리해 DB 커밋과 원자적으로 이벤트를 저장하고, 별도 스케줄러(500ms 주기)가 PENDING 이벤트를 폴링해 SQS로 발송한다. 발송 실패 시 exponential backoff(1s→2s→4s…최대 60s)로 최대 10회 재시도하며, 전부 실패 시 FAILED 상태로 기록해 수동 재처리 가능하다. SQS FIFO의 `messageDeduplicationId`를 jobId로 설정해 폴러 재시작/다중 인스턴스 상황에서도 중복 발송을 SQS 레벨에서 차단했다.
- 실패 웹훅 환불은 VIDEO_PROCESSING_REFUND:{videoId} 키로 멱등 처리해 중복 환불 차단

### 측정 방법
- `@SpringBootTest` 통합 테스트에서 20개 스레드를 `CountDownLatch`로 동시에 출발시켜 동일 videoId로 race condition 재현, 3회 반복 평균값으로 측정

### 결과
- 락/유니크/멱등 적용 전후의 Job 생성 수, 크레딧 이력, 상태 전이 결과 비교
- **데이터 정합성 개선**: 동일 videoId 동시 요청(20스레드)에서 중복 Job 생성이 평균 4.3건 → 1건으로 수렴, 중복 차감/중복 환불 위험을 구조적으로 차단
- **부분 실패 내구성 강화**: Outbox 도입으로 커밋 직후 장애가 나도 메시지 재전송이 가능한 운영 구조 확보
- **Trade-off**: 락/검증/보안 체크가 추가되어 평균 응답 시간이 소폭 증가할 수 있으나, 금전성 도메인에서 더 중요한 무결성과 운영 안정성을 우선 확보
- **운영 안정성 향상**: Redis 락은 선택적 성능 레이어로 설계되어 장애 시 자동으로 DB 비관락으로 fallback된다. 멱등 레코드는 DB에 저장되므로 Redis 장애와 무관하게 멱등성이 구조적으로 보장된다. Outbox 폴러 실패 시 exponential backoff(최대 60초)로 재시도하며, 최대 10회 재시도 후 FAILED 기록으로 메시지 유실을 방지한다.

## 면접 예상 공격 포인트
1. "Redis 장애 시 멱등성 어떻게 보장하나요?"
   → 멱등키가 Redis에만 있어서 Redis 죽으면 멱등성 붕괴 → 취약

2. "DB 락 fallback 어떻게 구현했나요?"
   → 미구현, 포폴 표현 수정 필요
   → "DB 비관락이 최종 정합성을 보장하는 구조로 설계"로 표현 변경 권장

3. "비관락이랑 유니크키 중복 아닌가요?"
   → 역할이 다름
   → 비관락: Video 상태 직렬화 (PENDING 검증)
   → 유니크키: 코드 버그/락 누락 시 최후 방어

4. "Outbox 폴러 재시도 시 중복은요?"
   → 미파악, 코드 확인 필요

5. "멱등키 만료 기간은요? 근거는?"
   → 준비 필요

6. "테스트 4.3건 어떻게 측정했나요? 스레드 몇 개?"
   → 준비 필요
